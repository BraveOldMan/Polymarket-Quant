# 链上机制与结算 — 技术规格

## 1. Conditional Token Framework（CTF）：split / merge / redeem

Polymarket 基于 Gnosis **Conditional Tokens Framework**。二元市场的 YES/NO 是 CTF 的 position token，
满足 **完整套件恒等式 `1 YES + 1 NO ≡ $1 USDC`**。回收抵押品只有两条路：**merge 完整套件** 或 **resolve 后 redeem**。

| 操作 | 输入 → 输出 | 用途 |
| --- | --- | --- |
| `split` | 1 USDC → 1 YES + 1 NO | 铸造完整套件（用于卖出两腿） |
| `merge` | 1 YES + 1 NO → 1 USDC | 合并完整套件赎回（**腿险兜底/锁利**） |
| `redeem` | 结算后的获胜 token → USDC | 市场 resolve 后赎付（见 §3 UMA） |

### 1.1 对策略① 单条件结构套利的修正
- **`Yes + No < 1`（如 0.97）**：在订单簿**买入 1 份 YES + 1 份 NO**（成本 0.97 + gas），再 **`merge` 赎回 \$1**，
  锁定 `1 − 0.97 − 成本` 的无风险利润。**不依赖"做空"**，merge 是确定性赎回。
- **`Yes + No > 1`（如 1.03）**：**`split` 1 USDC → YES+NO**，在订单簿**卖出两腿**收回 1.03，锁定 `1.03 − 1 − 成本`。
- **腿险兜底**：双腿只成交一腿时，用另一腿的链上 `split`/`merge` 对冲，或限价 + FOK 降低单腿暴露
  （执行细节见主文档 §5.6）。这把策略①从"两笔独立订单的腿险"降为"可确定性退出"。

## 2. NegRiskAdapter：互斥多结果事件的 Convert / Redeem

部分事件被组织为 **negative-risk set**（一组互斥结果，**至多一个** resolve YES）。`NegRiskAdapter` 在 CTF 之上
强制"恰好一个 YES"约束，并提供 **Convert**：把某市场的 **NO** 转换为**其余所有市场的 YES + 一定 USDC**。

### 2.1 对策略② 多条件互斥套利的修正
- 真实套利不只是"订单簿 `Σ Yesᵢ < 1` 时买全集"，还可用 **Convert/Redeem** 实现确定性收敛：
  - 当 `Σ Yesᵢ < 1`：买齐全集 → 事件结算后恰好一个 YES 赔付 \$1 → 锁定 `1 − Σ Yesᵢ − 成本`。
  - 利用 Convert 把零散 NO 头寸转换为等价 YES 组合 + USDC，降低多腿建仓成本与腿险。
- **前置校验（强制）**：该组必须是 NegRisk 事件且**穷尽 + 互斥**；漏项/重叠 = 伪套利（数据层 `MarketGroup`
  的 `is_exhaustive / is_mutually_exclusive` 校验，见数据层规格 §6）。

## 3. UMA 乐观预言机：结算时间线与争议风险

Polymarket 通过 `uma-ctf-adapter` 用 **UMA Optimistic Oracle** 结算。**到期 ≠ 即时结算**，流程：

```
事件结束 → 有人 propose 结果（质押 $750 PUSD 保证金）
        → 2 小时 liveness/争议窗
            ├─ 无争议：~2h 后结果可用 → 市场 resolve → 可 redeem
            └─ 有争议（对方质押等额保证金）：升级 DVM 投票 → 总计约 4–6 天 → 结果可能翻转
```

> 数字为撰写时核实的当期口径（保证金/奖励/时长），**实现时须复核**——UMA/Polymarket 参数会调整。

### 3.1 工程含义（关键）
- **资金锁定**：从建仓到可 redeem 之间资金被锁定；争议会把锁定从 ~2h 拉长到 4–6 天，**必须计入 RC 占用**
  与机会成本（主文档 §9.8）。
- **争议翻转风险**：临到期看似"已定"的结果可能被 propose 错误或被争议后翻转——这正是策略⑤"卖 No"的
  **steamroller 尾部**与"语义模糊"（如规则边界争议）的真实载体。
- **对策略⑤ 的修正**：时间衰减收敛假设必须叠加 UMA 时间线：收益兑现点是 **redeem 可用时刻**，不是事件结束时刻；
  争议概率高/规则模糊的市场（数据层 `ExpiringTarget.settlement_rule` 非清晰）**转人工**或不做。

## 4. 费用、Gas 与代理钱包

| 项 | 说明 |
| --- | --- |
| 交易费 | Polymarket CLOB 当前撮合费率以**官方当期费率表为准复核**；文档 `fee` 为占位，不假设为 0 |
| Gas | Polygon（chain_id=137）上 `split/merge/redeem/convert` 与结算交易的 Gas，按 `gas_bps` 计入 EV（主文档 §4.7/§11.4） |
| USDC 授权 | 交易/CTF 操作前需 USDC `approve`；allowance 监控与**最小化授权**（主文档 §11.5 热钱包安全） |
| 代理钱包 | Polymarket 用代理钱包（proxy / Gnosis Safe 类）；`py-clob-client` 的 `signature_type` 与 `funder` 据此配置 |

## 5. 链上执行安全（P2）

- **MEV / 抢跑**：`split/merge/redeem/convert` 是公开 mempool 交易，可能被抢跑/夹击；
  对策：私有交易通道（如 Polygon 的私有 RPC/Flashbots 类）、滑点与最小收益保护、必要时拆单。
- **Nonce / 重放**：交易 nonce 管理与幂等键（`intent_id`）联动，崩溃恢复后不重复发链上交易（主文档 §5.5/§7）。
- **失败重试**：链上确认失败按退避重试，超阈升级告警并进入对账冻结（主文档 §8）。
