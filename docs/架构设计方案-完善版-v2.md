## 0. 免责声明与口径说明（先读）

在进入架构之前，必须澄清以下口径，避免把营销表述当作工程依据：

- **收益数字与案例仅为示意，非承诺。** 两份来源稿中的"48 小时 1322% 收益""Coan2 累计 97.5 万美元"
  "SK639 用 286 美元做到 21 万、胜率 98.7%""Coinman2 基于 28 个代码库""单笔获利 5.9 万美元"等，
  均为来源自述的模拟/预估/第三方传闻，**不构成本系统的性能目标或收益保证**。真实表现取决于流动性、
  滑点、Gas、对手盘与监管环境，须以实盘小资金灰度结果为准。
- **市场机会规模可能被夸大（数据注水）。** 来源提到的"3900 万美元年度套利机会"存在 **Data Inflation**：
  部分"价差"由**洗盘交易 / 做市商自成交（Wash Trading）**人为制造深度假象（如 0.036 买入后在 0.98 处
  自挂大单接盘）。系统必须内建**洗盘检测**并把疑似自成交市场剔除出可交易集（见 §9.6）。
- **高胜率 ≠ 低风险（负偏尾部）。** 时间衰减"卖 No"类策略（SK639 式）是典型的
  **"在压路机前捡硬币"**：胜率极高但单次黑天鹅可吞掉长期累积利润。必须用尾部限额、单事件上限与
  熔断约束，不可因高胜率放大杠杆（见 §1.2 与 §9.3）。
- **流动性陷阱。** 冷门市场即使方向正确，也可能因缺乏接盘者而无法在结算前了结。开仓前须做**出场流动性
  预检**（见 §9.6）。
- **跟单/链上情报的合规边界。** 跟踪"高胜率/内幕"钱包、为空投而刷地址活跃度、规避监控的"钱包轮换"等，
  涉及法律（内幕交易类）、平台 ToS 与税务风险。本系统**仅做公开链上数据的统计分析，不实现任何规避
  监控或身份隐匿机制**，相关策略默认进入人工合规复核（见 §1.2 策略⑥、§9.7）。
- **对手方/平台风险不可消除。** 资金托管于链上合约与 Polymarket 体系，存在**智能合约漏洞、平台规则突变、
  USDC 脱锚、Polygon 桥/网络故障、预言机失效**等系统性风险，且多数无法通过分散下注对冲。须设平台级敞口
  上限与平台异常熔断（见 §9.9）。
- **合规风险须独立评估。** 预测市场在部分司法辖区受限或禁止，涉及 KYC、地域限制与衍生品监管。
  系统上线前需就**部署地域、账户合规、资金来源**单独做法律评估。本文档不构成合规/投资建议。
- **本系统不实现任何规避 AI 安全对齐的能力。** 两份原稿中的 "G-Mode（无审查接口 / 绕过安全对齐处理
  敏感政治论点）" 已被**移除**，替换为「敏感主题的合规处理与人工复核」机制（见 §4.8）。
- **风险自负。** 量化套利可能在极端行情、合约漏洞、预言机故障或链拥堵下产生超额亏损。所有自动化
  下单路径都必须有仓位上限、熔断与人工兜底（见 §9）。

## 1. 系统战略定位与套利策略族

### 1.1 定位
核心盈利逻辑不是"方向性预测"，而是**系统性套利**：通过全维度数据解算资产的「公允概率 / 公允价值」，
与 Polymarket 当前定价对比，捕捉「价格空隙」（Price Gaps），以模块化、可监控、风险受控的方式执行。

Polymarket 的可套利性来自三个内生缺陷：**订单簿碎片化**（流动性时空分布不均）、**预言机延迟**
（现实事件 → 链上价更新的时差）、**语义模糊性**（规则边界模糊导致认知偏差与定价失衡）。
二元代币遵循 **收敛至结算价值**：事件发生 → 收敛到 \$1，否则归零；订单簿价即市场实时隐含概率
（如 \$0.65 = 65%）。

### 1.2 套利策略族（6 类）

> 每类策略都是可独立启停的模块（CLAUDE.md「热替换」），共享数据/校验/风控基础设施。
> 风险大致递增：① 结构性/低风险，② 中（多腿/流动性），③ 偏高（语义幻觉），④ 中（时间对齐），
> ⑤⑥ 高风险且涉合规/尾部，需更强约束。**注意：所有策略仍承担 §9.8 结算与 §9.9 平台/对手方系统性风险。**

**① 单条件结构套利（Single-Condition Hedge，经 CTF 实现）**
- 机制（真实）：基于 CTF 完整套件恒等式 `1 YES + 1 NO ≡ $1`。`Yes+No < 1`（如 0.97）买两腿 →
  **`merge` 赎回 \$1** 锁定 `1 − (Yes+No) − 成本`；`Yes+No > 1` 则 **`split` \$1 → 卖两腿**。
  不依赖"做空"，merge/split 是确定性赎回（详见 [链上机制与结算](链上机制与结算-技术规格.md) §1）。
- 风险：近无风险；主要是滑点/Gas/腿险（单腿成交）——**腿险用 merge 兜底**可确定性退出。
- 数据：单市场订单簿快照（`OrderBookSnapshot`）。

**② 多条件互斥套利（Multi-Condition，经 NegRiskAdapter 实现）**
- 机制（真实）：一组**互斥且穷尽**的结果（NegRisk 事件，至多一个 YES）。`Σ Yesᵢ < 1` 时买全集 →
  结算时恰好一个 YES 赔付 \$1，锁定 `1 − Σ Yesᵢ − 成本`；可用 **NegRiskAdapter Convert** 把零散 NO
  转为其余市场 YES + USDC，降腿险（详见 [链上机制与结算](链上机制与结算-技术规格.md) §2）。
- 风险：**必须校验是 NegRisk 事件且穷尽+互斥**（漏项/重叠 = 伪套利）；多腿执行与流动性。
- 数据：同一事件的全部互斥市场组（Gamma API 事件分组 → `MarketGroup`）。

**③ 跨市场语义映射套利（Cross-Market Semantic Mapping）**
- 触发：用 LLM 识别**本质等价但表述不同**的市场（如"共和党胜选" vs 各州/百分比子市场组合），
  当等价组合价格与主市场**脱钩**时对冲。
- 风险：**语义等价判断的幻觉风险最高** —— 必须人工/规则二次确认等价关系，置信不足不开仓。
- 数据：市场元数据 + LLM 语义映射（见数据层 §6 信号 + 推理层 §4.3 证据融合）。

**④ 延时套利（Latency Arbitrage）**
- 触发：用 Binance BTC/ETH 短周期价算公允概率 `q`，抢在 Polymarket 隐含概率 `p` 调整前下单（见 §4.2a）。
- 风险：时间对齐失败/数据陈旧（用 `Δ_max` 约束，见 NFR §11.1）。
- 数据：Binance WS + Polymarket WS 时间对齐。

**⑤ 时间衰减趋势套利（Time-Decay，"卖 No"，SK639 式）** —— **高风险，负偏**
- 触发：临到期（结算前 3–5 天）针对**极端目标**（如"BTC→\$150k"）在概率 2%–8% 处买入 **No**，
  赚概率随时间向 \$1 收敛的剩余时间价值。
- **UMA 时间线修正（关键）**：收益兑现点是 **redeem 可用时刻**，不是事件结束时刻——UMA 提案 + 2h 争议窗，
  **有争议则 4–6 天且结果可能翻转**；收敛假设必须叠加该时间线与资金锁定（见 §9.8 与
  [链上机制与结算](链上机制与结算-技术规格.md) §3）。
- 风险：**"在压路机前捡硬币"** —— 胜率极高（来源称 98.7%，仅示意）但**单次黑天鹅/争议翻转吞噬长期利润**。
  必须：单事件上限、**VaR/CVaR 尾部限额**、禁止因高胜率加杠杆、临到期事件级熔断（见 §9.3）；
  规则模糊市场转人工或不做。
- 数据：临近到期市场筛选 + 极端目标识别 + 到期/结算规则解析（`ExpiringTarget`）。

**⑥ 聪明钱跟单（Copy-Trading / 钱包情报）** —— **高风险，涉合规**
- 触发：链上识别**高胜率/疑似内幕**钱包（统计胜率、单笔盈亏、损耗、"新活跃且高胜率"），跟随其建仓方向。
  由于结算价固定为 \$1，跟单不改变结算（无 PVP 抢跑稀释）。
- 合规红线：**仅统计公开链上数据**；不实现"钱包轮换规避监控""为空投刷活跃度"等机制；跟随疑似内幕
  地址可能涉法律风险，**默认进入人工合规复核**（见 §9.7）。
- 数据：Polygon 链上钱包画像（PnL/胜率/活跃度），见数据层 §6 + §9.7。

| 策略 | 触发 | 难度/风险 | 关键数据 |
| --- | --- | --- | --- |
| ① 单条件 | Yes+No≠1 | 低 | 单市场订单簿 |
| ② 多条件互斥 | ΣYesᵢ≠1 | 中 | 事件互斥市场组 |
| ③ 跨市场语义 | 等价组合脱钩 | 高（幻觉） | LLM 语义映射 |
| ④ 延时 | q vs p 时差 | 中 | Binance+Polymarket 对齐 |
| ⑤ 时间衰减卖No | 临到期极端目标 2–8% | **高（尾部）** | 到期筛选+规则解析 |
| ⑥ 聪明钱跟单 | 高胜率钱包建仓 | **高（合规）** | 链上钱包画像 |

### 1.3 设计哲学
把 AI 从"被动分析工具"升级为具备**眼睛（数据）、大脑（推理）、手脚（执行）**的闭环交易实体，
但每个环节都有**校验、共识与熔断**，而非无约束的全自动黑箱。

## 3. 总体架构与数据流

六层构成闭环：**数据（眼睛）→ 推理（大脑）→ 编排（手脚）**，由**基础设施层**承载、**结算层**落地、
**风险监控层**贯穿全程。

```
                         ┌─────────────────────────────────────────┐
                         │        L6 风险监控层（贯穿全程）          │
                         │  限额 / 熔断 / Alpha 衰减 / 实时仪表盘   │
                         └─────────────────────────────────────────┘
   L3 数据与信号层（眼睛）        L1 推理层（大脑）        L2 编排执行层（手脚）
 ┌───────────────────────┐   ┌──────────────────┐   ┌────────────────────────┐
 │ Polymarket WS/REST    │   │ Claude 主策略师  │   │ 多空辩论 → Risk 否决   │
 │ Binance WS（锚点）    │──▶│ Claude Squad分域 │──▶│ → 定仓 → CoT 落库      │
 │ Polygon 链上监控      │MCP│ 公允概率 vs 市价 │   │ → py-clob-client 下单  │
 │ OpenBB 宏观/基本面    │   │ 期望值判定       │   │                        │
 │  ↓ pydantic Schema 校验│   └──────────────────┘   └───────────┬────────────┘
 │  ↓ 归一化事件 + 信号  │                                       │
 └───────────────────────┘                          L4 基础设施 / L5 结算层
                                                  容器 / 总线 / 密钥 / 钱包签名 / 对账
```

**硬边界（来自 CLAUDE.md 约束）：**
1. **未过 Schema 校验的数据不得进入推理层**（防"幻觉阿尔法"）。Schema 校验仅防**格式**；对**内容字段**
   （市场标题/规则、新闻、SEC、链上 memo 等不可信文本）须额外做 **Prompt 注入防御**：内容与指令分离、
   转义/分隔、注入特征检测、工具白名单（见 §11.5；数据层规格 §1）。
2. **策略/采集模块可热替换**：逻辑参数化、配置驱动，便于在 Alpha 衰减时切换。
3. **开仓必须走「多空辩论 + 风险一票否决」共识流，并落 CoT 日志**，不存在绕过共识的直连下单路径。
4. **不实现规避安全对齐的逻辑**（G-Mode 已移除）。
5. **关键状态持久化、可崩溃恢复**：编排状态机与未结算订单**事件溯源 + 幂等重放**，进程崩溃/重启不丢、
   不重复下单/不重复发链上交易（见 §5.5、§7）。

## 4. 第一层：AI 推理层（系统大脑）

### 4.1 职责
计算资产「真实/公允概率」，与 Polymarket 隐含概率（市价）做对比，识别**正期望值**机会并生成决策意图。

### 4.2 公允概率建模方法

> 总原则：**推理层输出概率必须带置信区间，禁止点估**；不同市场类型用不同估计器，统一归一到
> `FairValueEstimate(fair_prob, conf_lo, conf_hi, basis, inputs_ref)`（见数据层规格 §4）。

**(a) 加密阈值型市场**（如"BTC 在 T 时刻 ≥ X"）—— 以 Binance 价为锚的解析估计：

- 设当前价 `S₀`（Binance 现货/合约中间价），到期剩余 `τ = (T − now)`（年化），波动率 `σ`
  （优先用期权隐含波动率，缺失时回退到已实现波动率 EWMA，见下）。
- 对数正态近似下，价格 `S_T` 超过阈值 `X` 的概率：

  ```
  q = Φ( ( ln(S₀/X) + (μ − σ²/2)·τ ) / (σ·√τ) )      # μ 默认取 0（风险中性近似，可配置）
  ```
  其中 `Φ` 为标准正态 CDF。"≤ X"取 `1 − q`。

- **波动率估计**：已实现波动率用 EWMA：`σ²_t = λ·σ²_{t-1} + (1−λ)·r²_t`，`r_t` 为对数收益，
  `λ` 默认 `0.94`（可配置）；优先用隐含波动率，二者差异过大时降低置信（拉宽区间）。
- **升级：期权隐含分布（首选，短久期更准）**：用 **Deribit BTC/ETH 期权**的隐含波动率曲面/风险中性密度
  （OpenBB `deribit` provider）直接得到"到期超阈概率"，比对数正态+EWMA 更贴合真实分布（含偏度/肥尾）。
  对数正态作为**回退/交叉校验**；两者背离大时拉宽 `[q_lo,q_hi]`。
- **漂移修正**：`μ=0` 风险中性近似可用**永续 funding/期现基差**修正，减少系统性偏差。
- **置信区间**：对 `σ` 做扰动 `σ·(1 ± k_σ)`（`k_σ` 默认 `0.15`）得到 `[q_lo, q_hi]`；样本不足或数据陈旧时进一步拉宽。
- **适用边界**：仅用于有明确数值锚的市场；强假设（对数正态、σ 平稳）在临近到期/极端行情下失效，
  须由置信区间体现并由风控降权。

**(b) 事件型市场**（政治/体育/社会）—— 证据加权 + LLM 主观概率：

- 无价格锚，由 Claude 基于结构化证据集给出主观概率与区间，**强制输出 `[q_lo, q_hi]` 与依据 `inputs_ref`**。
- 历史可回测的子类（如体育）引入基准模型（如 Elo/隐含赔率）作为先验，LLM 仅做调整，降低幻觉。

### 4.3 证据融合与冲突消解

- 输入为数据层归一化事件：行情（`PriceTick`）、链上（`ChainEvent`）、宏观（`MacroSignal`）、基本面摘要。
- **融合**：按来源可靠性与时效赋权，对独立证据做概率融合（如对数几率加权平均）：
  `logit(q) = Σ wᵢ·logit(qᵢ) / Σ wᵢ`，权重 `wᵢ` 随数据新鲜度衰减。
- **冲突消解**：当强证据方向相反（如链上大额买入 vs 宏观利空）时，**不强行取均值**——拉宽置信区间并标记
  `conflict=true`，由编排层在辩论中重点对抗；冲突过大则不产出信号。
- **可追溯**：每个 `q` 必须记录参与融合的事件 id 列表（`inputs_ref`），供 CoT 与衰减分析回溯。

### 4.4 置信区间与概率校准

- 所有概率输出附 `[q_lo, q_hi]`；**触发开仓要求区间不跨越市价 `p`**（见 §4.7）。
- **离线校准监控**：定期计算 Brier score 与可靠性曲线（预测概率 vs 实际频率分桶），评估模型是否系统性高/低估。
- 校准漂移（可靠性曲线偏离对角线超阈）作为 **Alpha 衰减信号之一**回流风控（§9.4），触发参数自适应或人工复核。

### 4.5 Claude 调用契约

| 项 | 约定 |
| --- | --- |
| 模型选型 | 主策略/终判用 `claude-opus-4-8`（深度推理）；分域初筛/证据摘要用 `claude-haiku-4-5`（控本提速） |
| 输出格式 | **结构化 JSON**，按 `FairValueEstimate` schema 约束；解析失败即重试/丢弃，不进入下游 |
| Prompt 结构 | system（角色+约束+输出schema）/ 证据块（MCP 注入的类型化数据）/ 任务指令；证据与指令分离 |
| 温度 | 概率估计用低温（默认 `0.2`）求稳定可复现；辩论 Agent 可略高以产生差异化论点 |
| 重试 | 指数退避；JSON 校验失败重试上限 `2`，超限丢弃并告警 |
| 限流 | 统一任务队列 + 令牌桶（按模型分别限流），避免触达 Rate Limit |
| Squad 分域 | `politics/crypto/sports/macro` 各自维护领域提示词与上下文；单域不可用回退通用实例并在仪表盘标降级 |
| 安全 | 严守模型安全对齐；**不构造规避对齐的提示或接口**（G-Mode 已移除，见 §4.8） |

### 4.6 抗 Alpha 衰减（替代 C3 Coder）
- **监控**：实盘命中率、价差收敛速度、拥挤度（同向地址增多）、校准漂移（§4.4）。
- **响应（分级、可控）**：
  1. 参数自适应：在配置层调整阈值 `θ`、仓位上限、超时（**热更，无需改代码**）。
  2. 策略切换：启用/停用模块（模块化热替换）。
  3. 离线再训练/再设计：由**人工审核**后合入，**禁止 AI 自动改写并直接上线生产代码**。

### 4.7 EV 与仓位算法（受约束 Kelly）

- **EV 口径（含成本）**：买 YES 净边际 `edge = q − p`，买 NO 为 `p − q`；
  `edge_net = edge − (fee + slippage + gas_bps)`（成本项见 NFR §11.4；`fee`/Gas 按当期 Polymarket 费率表与
  Polygon Gas 复核，见 [链上机制与结算](链上机制与结算-技术规格.md) §4）。
- **触发门槛**：`edge_net ≥ θ`（`θ` 默认覆盖成本后再留安全边际，默认 `θ = 0.02`，可配置），
  且置信区间不跨越 `p`（`q_lo > p` 买 YES 或 `q_hi < p` 买 NO）。
- **受约束 Kelly 仓位**：二元下注 Kelly 比例 `f* = (q·b − (1−q)) / b`，`b` 为净赔率（由 `p` 与成本推出）。
  实际下注比例：
  ```
  f = clip( κ · f* · shrink, 0, f_max )
  ```
  - `κ` = 分数 Kelly 系数（默认 `0.25`，抑制过度下注）；
  - `shrink` = 按置信区间宽度收缩（区间越宽越小，如 `1 − (q_hi − q_lo)/w_ref`）；
  - `f_max` = 单笔仓位上限（风控 §9，默认占可用风险资本 `2%`）。
- **冲突/低置信**：`conflict=true` 或区间过宽时 `shrink→0`，即不开仓。

### 4.8 敏感主题处理（替代 G-Mode）
- 政治/社会等敏感事件：推理可参与，但**开仓决策强制进入人工复核通道**，不自动执行。
- 严守模型安全对齐，不构造规避对齐的提示或接口。

## 5. 第二层：编排与智能体执行层（动作中枢）

### 5.1 「多空辩论 + 风险一票否决」状态机

将开仓流程实现为显式状态机，每步有明确输入/输出契约（详细字段见配套技术规格）：

```
PROPOSED        ← 推理层产出 TradeIntent(market, side, q, p, ev, confidence)
   │
DEBATE          ← Bull Agent / Bear Agent 基于同一证据集对抗论证
   │              产出 DebateResult(bull_case, bear_case, winner, residual_risk)
   │
RISK_REVIEW     ← Risk Agent 检查敞口/限额/相关性/事件黑名单
   │              VETO → REJECTED（记录原因）；PASS → 继续
   │
SIZED           ← 计算开仓比例（受约束 Kelly + 限额裁剪）
   │
COT_LOGGED      ← M-Thinker：结构化 CoT 决策轨迹落库（审计必需）
   │
EXECUTED        ← Trading Agent 经 py-clob-client 下单；写入订单/成交事件
```

- **任何 REJECTED/VETO 都必须留痕**，进入风险监控层统计。
- **幂等**：每个 TradeIntent 有唯一 id，重试不重复下单。

### 5.2 消息契约（pydantic 风格示意）

每个状态转移的输入/输出都是显式类型化对象，落库可回溯：

```python
class TradeIntent(BaseModel):          # 推理层 → 编排层
    intent_id: str                      # 幂等键（uuid）
    condition_id: str
    side: Side                          # YES / NO
    market_prob: Decimal                # p（下单时刻隐含概率）
    fair_prob: Decimal                  # q
    conf_lo: Decimal; conf_hi: Decimal  # 置信区间
    edge_net: Decimal                   # 扣成本净边际
    basis: str                          # latency_arb / structural / macro / event
    inputs_ref: list[str]               # 证据事件 id（可追溯）
    domain: str                         # politics/crypto/sports/macro
    created_ts: datetime

class DebateRound(BaseModel):
    round_no: int
    bull_case: str                      # 看多论点（基于同一证据集）
    bear_case: str                      # 看空论点
    bull_conf: Decimal; bear_conf: Decimal

class DebateResult(BaseModel):
    intent_id: str
    rounds: list[DebateRound]
    winner: str                         # bull / bear / inconclusive
    divergence: Decimal                 # 最终分歧度（用于终止判定）
    residual_risk: list[str]            # 未消解的风险点
    sensitive_flag: bool                # 命中敏感主题→人工通道

class RiskDecision(BaseModel):
    intent_id: str
    decision: str                       # PASS / VETO
    veto_reasons: list[str]             # 命中的检查项（见 5.4）
    exposure_snapshot: dict             # 当时各维度敞口
    ts: datetime

class SizedOrder(BaseModel):
    intent_id: str
    fraction: Decimal                   # f（受约束 Kelly，§4.7）
    notional: Decimal                   # 名义金额
    max_slippage: Decimal
    time_in_force: str

class CoTRecord(BaseModel):             # 审计落库（§5.5）
    intent_id: str
    intent: TradeIntent
    debate: DebateResult
    risk: RiskDecision
    sizing: SizedOrder | None
    final_state: str                    # EXECUTED / REJECTED
    model_meta: dict                    # 各步用的模型/温度/token
    ts: datetime
```

### 5.3 辩论协议
- **同源证据**：Bull/Bear 接收**完全相同**的证据集与 `TradeIntent`，仅角色 prompt 不同，避免信息不对称。
- **轮次**：上限 `N`（默认 `2`）。终止条件任一满足即停：
  1. **收敛**：两侧概率估计差 `|bull_conf − bear_conf| ≤ ε`（默认 `0.05`）；
  2. **分歧过大**：`divergence ≥ d_max`（默认 `0.4`）→ 判 `inconclusive`，**不开仓**；
  3. **超时**：超过辩论时间预算（NFR §11.1）。
- **裁决**：由主策略实例（opus）综合两侧给出 `winner`；`inconclusive` 或 `sensitive_flag` → 不进入下单（后者转人工）。

### 5.4 一票否决规则（Risk Agent）
进入 `RISK_REVIEW` 后逐项检查，**命中任一即 VETO**并记录原因：

| 检查项 | VETO 触发条件 |
| --- | --- |
| 敞口/限额 | 开仓后超过单市场/单领域/总敞口上限（§9.1） |
| 相关性 | 与现有持仓同因子聚类后合并敞口超限（§9.1） |
| 事件黑名单 | 命中敏感/受限事件清单（→ 人工通道） |
| 数据新鲜度 | `inputs_ref` 中关键证据 `recv_ts` 超时（latency_arb 用 Δ_max，见数据层规格 §5） |
| 置信/边际 | `edge_net < θ` 或置信区间跨越 `p`（与推理层不一致时以风控为准） |
| 资金 | 可用风险资本不足 / USDC 余额或 allowance 不足 |
| 市场质量 | `wash_suspect=True`（疑似洗盘）或 `exit_liquidity_ok=False`（出场流动性不足），见 §9.6 |
| 互斥完备性 | 多条件套利组未通过穷尽/互斥校验；跨市场语义等价非 rule/human 确认（§1.2②③） |
| 合规闸门 | 跟单命中 `insider_suspect` 或敏感事件，须人工复核（§9.7、§4.8） |
| 结算风险 | UMA 资金锁定超限 / 结算规则模糊 / 临近争议期（§9.8） |
| 平台风险 | 平台级敞口超限 / 平台规则变更 / USDC 脱锚 / 预言机异常（§9.9） |
| 熔断态 | 系统处于熔断或 kill-switch（§9.3、§9.5） |

- VETO/REJECTED 全部产出 `RiskDecision` 并落库，回流风控统计（§9.4）。

### 5.5 CoT 审计与失败处理
- **CoT 落库**：每个 intent 无论成败都写一条 `CoTRecord`（§7 存储），供回溯、复盘、合规审计与衰减分析。
- **幂等**：`intent_id` 唯一；下单前查重，重试不重复下单。
- **状态超时与补偿**：每个状态有超时预算（NFR §11.1），超时则置 `REJECTED(timeout)`；
  已下单未确认的走结算层重试/撤单补偿（§8）。
- **持久化与崩溃恢复**：状态机每次转移**事件溯源落库**（§7 存储），进程重启从持久状态**幂等重放**恢复；
  对"已发链上交易"用 `intent_id` + nonce 去重，**崩溃后不重复下单/不重复 split/merge/redeem**
  （见 [链上机制与结算](链上机制与结算-技术规格.md) §5）。
- **回流**：REJECTED/VETO 计数进入风控，用于熔断与 Alpha 衰减判定。

### 5.6 执行器（Trading Agent）
- 基于官方 `py-clob-client`：`ClobClient(host, key, chain_id=137, signature_type, funder)`，
  `create_or_derive_api_creds()` 派生 API 凭证。
- 支持市价单（`MarketOrderArgs` + `create_market_order` + `post_order`）与限价单；
  下单后订阅 WS 跟踪成交/撤单/结算。
- **权限最小化（替代 Superpers）**：执行 Agent 仅持有下单所需的白名单能力，密钥经密钥管理服务注入。
- **maker vs taker 成本**：taker 吃单付价差、maker 挂单可能拿优惠但有不成交风险；高频价差策略需权衡，
  把执行方式计入 `edge_net`。
- **多腿近原子执行（策略①②）**：双/多腿尽量同批提交并设 FOK/FAK 降低单腿暴露；若仍单腿成交，
  立即用 **CTF merge / NegRisk convert 兜底**确定性退出（见 [链上机制与结算](链上机制与结算-技术规格.md) §1.1/§2.1）。

## 9. 第六层：风险监控层（贯穿全程）

> 所有阈值均为**默认值、可配置、待灰度校准**；以"占可用风险资本（RC）的百分比"表述，避免绑定绝对金额。
> `RC` 定义见 §9.2。

### 9.1 限额体系

| 维度 | 默认上限 | 说明 |
| --- | --- | --- |
| 单笔仓位 `f_max` | RC 的 **2%** | 与 §4.7 Kelly 裁剪一致 |
| 单市场敞口 | RC 的 **5%** | 同一 `condition_id` 累计 |
| 单领域敞口 | RC 的 **20%** | `politics/crypto/sports/macro` 各自 |
| 总敞口 | RC 的 **60%** | 预留缓冲应对滑点与未结算 |
| 相关性敞口 | 合并后视同单市场限额 | 见下 |

- **相关性敞口合并**：把高相关事件聚为一个因子簇（如同一选举的多个关联市场、同一标的的多档阈值市场），
  按相关系数 `ρ` 加权合并敞口：`E_cluster = Σ Eᵢ·max(ρ, ρ_floor)`（`ρ_floor` 默认 `0.3`），
  合并值受单市场限额约束，防止"分散开仓实则同一风险"。
- **升级：ρ 由结构估计而非手填**——构建**共结算图/因子模型**：同一事件、**同一 UMA 预言机问题**、同一底层标的
  视为强相关边；`ρ` 由历史共动（回测数据）+ 图结构推断，`ρ_floor` 仅作下限兜底。

### 9.2 资金管理（账户分层）

- **核心保证金**：维持账户运行的最低 USDC，不参与下注。
- **可用风险资本 RC** = 账户净值 − 核心保证金 − 未结算占用；所有 §9.1 百分比基于 RC 动态计算。
- **全局 Kelly 缩放** `κ`（§4.7，默认 `0.25`）：回撤升高时自动调小（见 §9.3），形成"亏损降杠杆"的负反馈。
- **单笔名义** = `f · RC`，再受 `max_slippage` 与对手盘深度裁剪（深度不足则减量或放弃）。

### 9.3 回撤与熔断（具体阈值）

| 触发器 | 默认阈值 | 动作 | 冷却 |
| --- | --- | --- | --- |
| 日内回撤 | 当日 PnL ≤ **−5% RC** | 暂停开仓，`κ` 减半 | 至次日或人工恢复 |
| 最大回撤 | 净值距峰值 ≤ **−15%** | 触发 kill-switch（§9.5） | 人工复核后恢复 |
| 单位时间亏损 | 滚动 1h 内亏损 ≥ **3% RC** | 暂停开仓 | 30 min |
| 连续失败 | 连续 **5** 笔 EXECUTED 亏损 | 暂停该领域开仓 | 人工/自适应 |
| 滑点异常 | 实际滑点 ≥ 预期的 **3×** | 暂停该市场，标记流动性恶化 | 动态 |
| 链拥堵 | Polygon Gas/确认延迟超阈 | 暂停下单，仅保留撤单 | 至恢复 |

- 熔断状态作为 Risk Agent 的 VETO 检查项之一（§5.4），下游不再产生新开仓。
- **尾部度量（负偏策略⑤重点）**：除仓位上限外，按组合维护 **VaR / CVaR（期望损失）** 限额——
  对"高胜率负偏"策略，破产风险来自尾部而非均值，CVaR 超限即拒新开仓（口径与回测见
  [回测与评估层](回测与评估层-技术规格.md) §5）。

### 9.4 Alpha 衰减量化（对接 §4.6）

| 指标 | 定义 | 衰减信号 |
| --- | --- | --- |
| 命中率 | 滚动窗 EXECUTED 盈利占比 | 低于基线 − k·σ |
| 价差收敛速度 | 信号出现到价差闭合的时长 | 持续变短（拥挤） |
| 拥挤度 | 同向地址/同向成交占比趋势 | 持续上升 |
| 校准漂移 | 可靠性曲线偏离对角线程度（§4.4） | 超阈 |

- 任一指标越界 → 触发 §4.6 分级响应（参数自适应 → 策略切换 → 人工再设计）。

### 9.5 kill-switch 与可观测性

- **kill-switch**：一键**停止所有开仓**，仅保留平仓/撤单；触发源包括最大回撤、对账差异、链异常、人工。
- **恢复**：须人工复核根因并解除，不自动恢复。
- **实时仪表盘**：TradingView Lightweight Charts 展示信号频率、动态损益曲线、各维度敞口、熔断/降级状态、
  各层健康（collector/推理/执行）。
- **事件黑名单**：敏感/高不确定事件进入人工通道（见 §4.8）。

### 9.6 市场质量过滤：洗盘与流动性陷阱（前置硬门槛）

针对来源稿揭示的 **Data Inflation / Wash Trading**（虚高的"3900 万机会"，见 §0），所有策略开仓前必须过：

- **洗盘/自成交检测**（数据层 `wash_filter.py`，§数据层 6.1）：识别自挂自接、关联地址环、极端价位大额
  接盘、深度与真实成交背离；命中市场 `wash_suspect=True` → **剔除可交易集**，作为 §5.4 VETO 项。
- **出场流动性预检**（数据层 `liquidity.py`，§数据层 6.2）：估算持有期/结算前以可接受滑点平仓所需深度；
  不足 → `exit_liquidity_ok=False` → 不开仓（防流动性陷阱）。
- **时间衰减尾部限额**（策略⑤）：单事件名义上限独立且更严（默认 ≤ RC 的 **1%**），临到期事件级熔断；
  **禁止因高胜率上调杠杆**——高胜率负偏策略的破产风险来自尾部而非均值。

### 9.7 跟单/链上情报的合规闸门（策略⑥）

- **仅公开数据统计**：钱包画像只用链上公开数据；**不实现身份关联、规避监控、钱包轮换跟踪用于隐匿、
  或为空投刷活跃度**等机制。
- **疑似内幕强制人工**：`WalletProfile.insider_suspect=True` 的跟随信号**不自动开仓**，进入人工合规复核；
  跟随疑似内幕地址可能涉法律风险（§0）。
- **VETO 联动**：合规闸门未通过即 §5.4 VETO 并留痕。

### 9.8 结算 / 预言机风险（UMA）

预测市场区别于普通交易所的核心风险：**到期不等于即时结算**（见 [链上机制与结算](链上机制与结算-技术规格.md) §3）。

| 风险 | 控制 |
| --- | --- |
| 资金锁定 | 建仓→可 redeem 之间资金锁定（无争议 ~2h，有争议 4–6 天）；**计入 RC 占用**与机会成本，限制临到期敞口 |
| 争议翻转 | 规则模糊/争议概率高的市场（`settlement_rule` 非清晰）→ 转人工或不做；设争议翻转风险敞口上限 |
| 预言机依赖 | UMA/适配器异常或参数变更 → 暂停依赖结算的新开仓，已有头寸优先平仓 |
| 时间线建模 | 策略⑤等收敛类必须把 UMA 时间线建进收益兑现点与回测（见回测文档 §3） |

- 上述作为 **§5.4 新增 VETO 项「结算风险」**：资金锁定超限 / 规则模糊 / 临近争议期 → VETO。

### 9.9 平台 / 对手方风险（系统性，不可分散）

预测市场套利的"无风险"仅指**事件结果维度**；资金始终承担平台与基础设施的系统性风险，须单列管理：

| 风险 | 控制 |
| --- | --- |
| 智能合约漏洞 | CTF/NegRisk/Exchange 合约风险 → **平台级总敞口上限**（默认 ≤ 账户净值的可承受比例，灰度期从严）；关注审计/事件 |
| 平台规则突变 | Polymarket 规则/费率/市场下架变更 → 监控公告，触发暂停相关市场新开仓 |
| USDC 脱锚 | 抵押资产脱锚 → 价格监控，偏离阈值触发降敞口/暂停 |
| Polygon 网络/桥 | 链拥堵、停块、桥故障 → 接 §9.3 链拥堵熔断；跨链资金保守 |
| 预言机失效 | UMA/适配器异常 → 见 §9.8，暂停依赖结算的新开仓 |
| 资金集中 | 单平台资金集中度上限；金库与交易热钱包分层（§11.5），仅必要资金在平台 |

- **平台异常熔断**：上述任一触发 → 收紧/暂停开仓，严重时 kill-switch（§9.5）；作为 §5.4「平台风险」VETO 项。

## 10. 分阶段路线图

> 策略按风险递增分期上线：**先低风险结构类（①④）+ 质量过滤（洗盘/流动性），后高风险类（⑤⑥）**，
> ⑤⑥ 须先建好尾部限额与合规闸门方可灰度。

| 阶段 | 范围 | 验收 |
| --- | --- | --- |
| **P1（当前重点）** | 数据与信号层最小闭环：Polymarket+Binance 采集 → Schema 校验 → **策略①④** 价差信号 + 洗盘/流动性前置过滤（**只读、不下单**） | 稳定产出经校验信号、洗盘市场被剔除、出场流动性预检生效，落日志+仪表盘 |
| P2 | 推理层接入：公允概率建模 + MCP 注入 + 期望值判定；加入**策略②**（互斥组校验） | 产出 q 与 EV 可解释；伪套利（漏项/重叠）被拒 |
| P3 | 编排层：辩论 + 风险否决 + CoT 落库（paper-trading）；加入**策略③**（语义映射，rule/human 确认） | 状态机全流程留痕、**状态持久化可崩溃恢复**；语义等价非纯 LLM 放行 |
| **P3.5（实盘前置门槛）** | **回测与评估层 + 纸面交易**：历史数据 + 成交/UMA结算延迟模型 + walk-forward + PnL 归因 + 校准（见 [回测与评估层](回测与评估层-技术规格.md)） | 各策略样本外收益/风险/校准可信，纸面交易端到端跑通 |
| P4 | 结算层灰度：小资金实盘 + CTF/NegRisk/UMA 结算 + 全套风控/熔断（见 [链上机制与结算](链上机制与结算-技术规格.md)） | 实盘**链上 CTF 余额**对账无差异，风控/熔断可触发 |
| P5 | 高风险策略灰度：**⑤ 时间衰减**（VaR/CVaR 尾部限额 + UMA 时间线）、**⑥ 跟单**（合规闸门+人工）；抗衰减与规模化 | 尾部/合规/结算约束生效、长期可监控可回溯 |

## 11. 非功能需求（NFR）

> 目标值均为**初始设定、可配置、待灰度校准**。延时套利对端到端延迟敏感，故延迟预算是一等需求。

### 11.1 延迟预算（端到端）

延时套利的核心是抢在 Polymarket 隐含概率调整前下单，端到端目标 **p95 ≤ 800ms**（信号产生→下单提交），分解：

| 环节 | p95 目标 | 备注 |
| --- | --- | --- |
| 采集→校验 | ≤ 30ms | pydantic 校验 + 归一化 |
| 信号计算 | ≤ 20ms | 公允价/价差，纯计算 |
| 推理（仅必要时） | ≤ 400ms | 加密阈值型走解析公式可**跳过 LLM**；事件型才调 Claude |
| 辩论（开仓前） | ≤ 250ms | 轮次/超时见 §5.3；高频套利可配置"轻辩论"或跳过 |
| 风控+定仓 | ≤ 20ms | 内存态检查 |
| 下单提交 | ≤ 80ms | py-clob-client 到 CLOB |
| **时间对齐 Δ_max** | **≤ 250ms** | Binance/Polymarket 事件 `recv_ts` 偏差上限（见数据层规格 §5），超限不开仓 |

- 纯数值套利路径应**绕过 LLM**以满足延迟；LLM 仅用于事件型/敏感判断，且不在最低延迟关键路径。

### 11.2 吞吐

| 指标 | 目标 |
| --- | --- |
| 并发监控市场数 | ≥ 200（按领域分片，可水平扩展） |
| 行情事件速率 | 峰值 ≥ 2,000 events/s（Binance+Polymarket 合计），背压不丢关键事件 |
| 信号产出 | ≤ 100ms 内完成单市场一轮信号评估 |

### 11.3 可用性 SLO 与降级

| 组件 | 可用性目标 | 降级/RTO |
| --- | --- | --- |
| 数据 collector | 99.9% | WS 断线自动重连，RTO ≤ 5s；重连期暂停依赖该源的信号 |
| 推理（Claude） | 99% | 主模型不可用回退轻模型/通用域；纯数值路径不依赖 LLM |
| 执行（CLOB） | 99.5% | 下单失败退避重试；超阈撤单并告警 |
| 整体开仓链路 | 尽力 | 任一关键依赖降级时**优先收紧风控/停开仓**，保平仓能力 |

### 11.4 成本护栏

| 成本项 | 控制 |
| --- | --- |
| Claude API | 分级用模型（opus 终判 / haiku 初筛）；纯数值路径不调 LLM；按日 token 预算 + 超额告警 |
| Gas（Polygon） | 单笔 `gas_bps` 计入 EV（§4.7），Gas 飙升触发链拥堵熔断（§9.3） |
| 基础设施 | 单节点起步，按吞吐水平扩展；成本超预算护栏告警 |

- **盈利护栏**：`edge_net` 必须为正（已扣 fee+slippage+gas），否则不开仓——成本直接进入下单决策。

### 11.5 安全与合规需求

- **密钥**：私钥/API key 绝不入库入码；环境变量 + 密钥管理（本地 .env / 生产 Vault/KMS）；执行 Agent 最小权限。
- **热钱包安全（最高价值目标）**：交易热钱包与**金库分层**（仅必要资金留在热钱包）；**签名服务隔离**
  （独立签名进程/HSM，业务进程不直接持私钥）；**出金白名单 + 单笔/日限额**；USDC **allowance 最小化**
  并定期回收；异常出金触发 kill-switch。
- **Prompt 注入防御**：喂入 LLM 的不可信内容（市场标题/规则、新闻、SEC、链上 memo）须内容/指令分离、
  转义分隔、注入特征检测；LLM 仅能调白名单工具，**不可由内容驱动触发下单/转账**（见 §3 硬边界#1）。
- **审计留痕**：CoT、RiskDecision、订单/成交、对账记录全量落库；保留期默认 ≥ **1 年**（可按合规调整）。
- **敏感事件**：政治/社会等走人工复核通道（§4.8、§5.4），不自动开仓；不实现规避安全对齐的逻辑。
- **合规前置**：上线前完成部署地域、账户 KYC、资金来源的法律评估（§0）；不满足则相关市场禁用。
