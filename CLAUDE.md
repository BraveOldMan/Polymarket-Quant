## 项目本质

面向 Polymarket 等去中心化预测市场的 **AI 量化套利系统**。核心盈利逻辑不是“方向性预测”，而是**系统性套利**：通过全维度数据解算出资产的「公允概率 / 公允价值」，与市场当前定价对比，捕捉瞬时「价格空隙」（Price Gaps），并以工业化、高频、确定性的方式执行。

延时套利（Latency Arbitrage）是关键手法之一：用币安 BTC/ETH 短周期合约数据计算公允价格锚点，抢在预测市场订单簿调整前下单。

## 六层架构（理解全局的关键）

系统是一个闭环：**数据（眼睛）→ 推理（大脑）→ 编排（手脚）**，外加底层基础设施、结算与全生命周期监控层。各层职责：

1. **AI 推理层（大脑）** — 计算「真实概率」并与市价做毫秒级对比，识别正期望值机会。
   - 主策略师为 Claude；`Claude Squad` 多实例并行分领域（政治/加密/体育），规避单模型 Context Window 与 Rate Limit 限制。
   - `C3 Coder` 监控 Alpha 衰减，在套利空间收窄时**自动重写策略代码**——这是抗策略拥挤、长期存续的核心机制，任何策略模块都应可被参数化/热替换。

2. **编排与智能体执行层（手脚）** — 把决策转为链上动作。
   - **「多空辩论 + 风险一票否决」决策流**：生成多头/空头智能体对抗辩论 → Risk Agent 按风险敞口行使否决权 → 仅在辩论胜出且风险受控时才确定开仓比例与时机。实现开仓逻辑时必须遵循此共识流程，不可单点直接下单。
   - `M-Thinker` 强制思维链（CoT）：开仓前必须生成可解释的逻辑文本日志，用于回溯与故障审计。

3. **数据与市场信号层（眼睛）** — 决定系统能力上限，类彭博终端的结构化采集，目标是**杜绝「幻觉阿尔法」**。
   - `OpenBB` 为数据主干（统一对接 100+ 数据源）；`DEXER` 解析 SEC 文件/财报；`Binance Collector` 提供价格发现锚点；`Crus` 监控 Polygon 链上鲸鱼钱包（领先信号）；`Bradop` 封装宏观指标（CPI/失业率）。
   - **MCP Server**：通过 Model Context Protocol 把类型化、已校验的金融数据 Schema 直接注入 Claude 上下文——所有进入推理层的数据都应经过 Schema 校验，防止格式错误导致推理偏离。

4. **基础设施层 / 5. 结算层 / 6. 风险监控层** — 设计文档中提及但未展开；实现时需补充设计。监控仪表盘建议用 Lightweight Charts 实时可视化信号频率与损益曲线。

## 编码时务必遵守的项目特有约束

- **数据必须经 Schema 校验后才进入推理层**（防幻觉阿尔法）——这是数据层的硬性边界，不要把未校验的外部数据直接喂给 LLM。Schema 只防格式：**自由文本（市场标题/规则、新闻、链上 memo）须做 Prompt 注入防御**，LLM 不可由内容驱动触发下单/转账。
- **策略模块要可热替换**：策略逻辑应参数化、模块化、配置驱动，避免硬编码。
- **开仓必须走多空辩论 + 风险否决共识流，并留下 CoT 日志**，不要实现绕过该流程的“直接下单”路径。
- **套利机制以链上为准**：策略①经 CTF `split/merge` 实现、②经 NegRiskAdapter `Convert`、结算经 **UMA 预言机**（到期≠即时结算，有 2h 争议窗、争议可达 4–6 天且可能翻转）。涉及①②⑤、结算、对账时先读 [docs/链上机制与结算-技术规格.md](docs/链上机制与结算-技术规格.md)。
- **状态机须持久化、可崩溃恢复**（事件溯源+幂等重放）；链上交易用 intent_id+nonce 去重，崩溃后不重复下单/split/merge/redeem。
- **热钱包是最高价值目标**：签名服务隔离、交易钱包与金库分层、出金白名单/限额、allowance 最小化。
- **任何策略实盘前须先过回测+纸面交易**（见 [docs/回测与评估层-技术规格.md](docs/回测与评估层-技术规格.md)）。
- 文档中的“G-Mode（无审查 AI 接口）”**已移除**；不实现规避安全对齐或规避监控/身份隐匿的逻辑，敏感事件走人工合规复核。
